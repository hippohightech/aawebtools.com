## 5. Dev.to / Indie Hackers Post

### Title

```
I Built 7 Free Online Tools With Zero Monetization Tricks -- Here's What I Learned
```

### Article (~800 words)

```
A few months ago I set out to build a simple TikTok downloader. I ended up with
seven tools, 150 pages in 8 languages, and a Docker stack I'm oddly proud of.
Here's the honest story.

## The Problem That Started It

I needed to download a TikTok video. I searched, clicked the first result, and
was immediately asked to create an account. The second result wanted my email.
The third had so many ads I couldn't find the actual download button.

This keeps happening with free online tools. They start free, add a signup wall,
then a usage limit, then a premium tier. The tool you bookmarked six months ago
now requires a $9/month subscription to do what it used to do for nothing.

I figured I could build something better. Not because I'm a better developer
than the people behind those tools, but because I don't need to squeeze revenue
out of every interaction.

## What I Built

AAWebTools (aawebtools.com) is a collection of 7 free tools:

1. **TikTok Video Downloader** -- paste a URL, get the video without watermark.
   Also handles slideshows and MP3 extraction.
2. **Twitter/X Video Downloader** -- same concept for tweets with video.
3. **Invoice Generator** -- 5 professional templates, bilingual (EN/FR),
   multi-tax support, instant PDF export.
4. **AI Content Detector** -- perplexity and burstiness scoring to flag
   AI-generated text.
5. **AI Text Humanizer** -- rewrites AI text to sound more natural, powered by
   Gemini 2.5 Flash.
6. **Pay Stub Generator** -- generates pay stubs for 9 countries (USA, Canada,
   UK, Australia, France, and others) with 3 PDF templates.
7. **Image Toolkit** -- compress, convert formats, resize, and add watermarks.
   All processing happens in the browser.

No signup. No accounts. No data stored. No premium tier.

## The Tech Stack

Everything runs in Docker on a single VPS:

- **Traefik** as the reverse proxy with automatic Let's Encrypt SSL
- **nginx** serving the static frontend
- **Node.js API** handling video downloads (using yt-dlp under the hood),
  invoice/pay stub PDF generation
- **Python API** for the AI tools (detector and humanizer)
- **Uptime Kuma** monitoring all endpoints with Telegram alerts
- **Umami** for privacy-respecting analytics (self-hosted, no cookies)

The entire frontend is static HTML/CSS/JS. No React, no Next.js, no build step.
I know that's unfashionable, but the pages load fast and I never have to debug a
hydration error.

## The Multilingual Rabbit Hole

The original plan was English and French. Then I thought, "How hard could 6 more
languages be?"

The answer: harder than building the tools.

I ended up with 8 languages (English, French, Spanish, German, Portuguese,
Arabic, Hindi, Indonesian) across roughly 150 pages. Every page has proper
hreflang tags pointing to every other language version. The blog articles are
translated individually, not just machine-translated and forgotten -- each has
its own localized JSON-LD schema markup.

Arabic required RTL layout support. Hindi required testing with fonts I'd never
worked with. Indonesian was the easiest, which I did not expect.

The tooling I built for this -- scripts to fix language selectors across all
pages, to inject hreflang tags, to validate cross-language links -- ended up
being more code than some of the tools themselves.

## What Actually Went Wrong

A lot.

**yt-dlp breaks constantly.** TikTok and Twitter change their APIs regularly.
The downloader that worked on Monday returns errors on Wednesday. I had to build
server-side proxy downloads with temp file cleanup because direct client-side
downloads kept getting blocked by CORS.

**PDF generation is painful.** Getting invoices and pay stubs to render
consistently across different data shapes, currencies, and tax systems took more
iteration than I want to admit. Five invoice templates sounds simple until you're
debugging why the French tax layout breaks when there are more than 12 line
items.

**AI detection is an unsolved problem.** My detector uses perplexity and
burstiness, which are legitimate signals, but they're not magic. Short text is
unreliable. Heavily edited AI text is hard to catch. I'm honest about that in
the tool's UI -- I'd rather under-promise than pretend I built something that
catches everything.

## What I'd Do Differently

**Start with monitoring.** I added Uptime Kuma and Telegram alerts late. I
should have set them up on day one. Knowing when something breaks before users
tell you is worth the 30 minutes of setup.

**Build the multilingual infrastructure first.** Retrofitting 8 languages onto
an existing site is brutal. If I were starting over, I'd build the i18n
framework before writing a single tool page.

**Write the blog earlier.** The blog drives more organic traffic than the tool
pages themselves. I waited too long to start publishing.

## What's Next

More tools. More languages. More blog content with original research (I have an
AI detector accuracy study in the works). And keeping everything free --
sustainably, through ads that don't ruin the experience.

If you want to try the tools: aawebtools.com

If something is broken, I genuinely want to know. I'm one person and I fix
things fast.

---

*Built by Karim Narimi / ScopeCove. Stack: Docker, Traefik, nginx, Node.js,
Python. Hosted on a single VPS.*
```
